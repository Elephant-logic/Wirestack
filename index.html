    Qutri Logic Engine
    Four-state reasoning for the app brain: verified, partial, unknown and conflicted.
    
          
    
            
            
            
          
    
        
    
        
    How to read Qutri: Verified is safe to build on, Partial needs tests or docs, Unknown needs investigation, and Conflicted must be repaired before deployment. This is not pass/fail — it is a confidence map for software understanding.
    
        
    
          
    Verified
    Evidence agrees and tests/validation support it.
    
          
    Partial
    Some evidence exists, but coverage is incomplete.
    
          
    Unknown
    Not enough evidence yet. Needs inspection or tests.
    
          
    Conflicted
    Signals disagree. Requires repair before trusting.

            User guide
            
    How to use WireStack Studio
    
            
    Use the Wizard for the normal path. Open Engines only when you need to inspect or tune the underlying systems.
    
          
    
        
    

        
    
          
    
            
    Recommended flow
    
            
      
              
    1. Wizard: start from an idea, guide/spec, or old project structure.
      2. 
              
    2. Generate blueprint: WireStack creates modules, wiring, context, and rules.
      3. 
              
    3. Choose provider: use Claude or OpenAI with the model ID you want.
      4. 
              
    4. Build: send the grounded blueprint context to the AI Build engine.
      5. 
              
    5. Verify: run validation, testing, Qutri scan, health score, and blueprint diff.
      6. 
              
    6. Export: save state, export manifest, or continue building.
      7. 
            
    
          
    
          
    
            
    Safety notes
    
            
    API keys: this is a browser prototype. Keys are used directly from this page. Use local/private testing only, or move provider calls behind a server proxy before public deployment.
    
            
    Model IDs: model names change. Use the editable model field if your provider has a newer model.
    
            
    Blueprint first: the safest loop is always understand → build → verify → diff.
